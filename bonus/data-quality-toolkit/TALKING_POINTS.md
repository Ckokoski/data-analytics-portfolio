# Talking Points — Data-Quality Audit Toolkit

A 2-minute interview walkthrough (STAR), plus three likely follow-ups — written in
my own words, grounded in what the tool actually does on the demo files.

## 2-minute walkthrough (STAR)

**Situation.** Every analysis starts with a file someone hands you, and the first
real question is never "what does it say?" — it's "can I trust this yet?" Missing
values, rows pasted in twice, the word "free" typed into a price column, an age of
215 — these slip past a quick glance and quietly corrupt every number downstream.
In five years of marketing technology I lived this from the production side: I
built QA checklists so a broken link or wrong merge field got caught *before* it
shipped to a million inboxes.

**Task.** I wanted that same QA instinct as code — a repeatable first-pass check I
could point at any CSV, that ends in a clear go/no-go call instead of a gut feel,
and that a non-technical manager could actually read. I built it as a self-directed
project to sharpen my Python.

**Action.** I wrote a Python CLI on pandas. It profiles any CSV — row and column
counts, per-column null rate, exact-duplicate rows and duplicate keys, columns
that are numeric-but-for-a-few-stray-text-values, and out-of-range numbers against
simple configurable rules. Two design choices I'd call out: I read the whole file
as text first so the *tool* decides what looks numeric, instead of letting pandas
silently mangle a column with one bad cell; and I gathered every PASS/REVIEW/FAIL
threshold into one named block at the top, so a reviewer can see exactly where I
drew the line and move it. It writes a plain-language Markdown report and ends in
one verdict with the specific reasons. To prove it works, I generate two synthetic,
clearly-labeled messy files — one that earns a FAIL, one a REVIEW.

**Result.** On the demo files it does exactly what I wanted: `messy_customers.csv`
returns **FAIL** — it catches a phone column that's 83% empty, 17% duplicate rows
that would double-count every total, and four repeated `customer_id`s that break the
join — while `messy_orders.csv` returns **REVIEW**, flagging text like "free" and
"two" sitting in number columns plus a 150% discount and a negative quantity. The
payoff is where I'd put it: the instant a CSV lands and before it touches anything
downstream — before a Power BI load, before a SQL import, before the first
`GROUP BY`. And because the exit code mirrors the verdict, the same command can gate
an automated pipeline, so a FAIL stops a load instead of quietly feeding a
dashboard. The instinct is the same one I built into email QA for five years —
catch the broken thing *before* it's in front of an audience — only now it's a
repeatable check anyone on a team can run and get the same answer.

## Likely follow-up questions

**1. "Is this real data?"**
Honest answer: the two demo files are **synthetic** — generated by a seeded script,
loudly labeled as fake, deliberately broken so the audit has real problems to catch.
The tool itself runs on any CSV. I made the demo data fake on purpose so I could
publish it; the *problems* in it aren't hypothetical, though. In marketing
technology I worked with the messy real-world version of this constantly — platform
exports and campaign CSVs where blanks, duplicate rows from a re-pulled report, and
"numbers" stored as text are the norm, not the exception. This tool is me taking the
manual checklist I already ran on that kind of file by eye and making it repeatable.

**2. "How does it decide PASS vs. REVIEW vs. FAIL?"**
Each finding is weighed: a serious problem — a column more than half empty, over 10%
duplicate rows, lots of out-of-range values — pushes toward FAIL; smaller blemishes
like a little missingness or a few duplicate IDs push toward REVIEW; nothing tripped
means PASS. All the thresholds live in one named block at the top of `audit.py` on
purpose, so the rules are visible and arguable, not buried.
For a *gate*, I'd keep the conservative defaults — a column 5% blank earning REVIEW
is a feature, because I'd rather be told and wave it off than miss it. The knob I'd
actually tune is the out-of-range rules, since "valid" is domain-specific, and those
already live outside the code in a JSON `--rules` file for exactly that reason. If I
were auditing a source that's intentionally blank in places — optional fields on a
form, say — I'd raise `NULL_RATE_REVIEW` at the top and leave a comment explaining
why, rather than teach myself to ignore a real warning. The point is I'd own that
number and be able to defend it, not inherit it silently.

**3. "Walk me through how it actually works."**
CSV in (read as text) → pandas answers one question per check → the findings collapse
into a verdict with written reasons → a Markdown report is written and a summary
printed, with the exit code mirroring the verdict so it can gate a pipeline. The
checks are small named functions, each unit-tested, plus an end-to-end smoke test on
the demo files.
The one I'd happily walk through line by line is `type_problems`, the
numeric-with-stray-text detector — it's what flags `free` and `two` in the orders
file. The trick is that I read every column as text first, then ask what *share* of
the non-blank cells could be read as a number (after stripping costume characters
like `$`, commas, and `%`, so `$1,250.00` still counts as numeric). If that share is
at or above 80% — the `NUMERIC_MAJORITY` threshold — I treat the column as one
that's *meant* to be numbers, and I surface the handful of cells that aren't, with
examples. That's deliberately the opposite of letting pandas guess: one `free` in a
price column would otherwise turn the whole thing into text and silently break every
sum downstream, so I'd rather the tool name the bad cells than hide them.
