# Talking Points — Excel Pivot Automation Engine

A 2-minute interview walkthrough (STAR), plus three likely follow-ups. The findings and decisions below are my own read of the included synthetic dataset, in my own words — not the tool's.

## 2-minute walkthrough (STAR)

**Situation.** Marketing and analytics teams constantly get handed raw campaign exports — one row per channel, per segment, per day — and they need the same summaries every time: spend and ROAS by channel, conversion rate by segment, the month-over-month trend, the best and worst performers. Doing that by hand in Excel is repetitive and error-prone.

**Task.** I wanted a tool that turns any campaign CSV into a clean, formatted, multi-sheet Excel report in seconds — and I built it as a self-directed project to sharpen my Python while solving a problem straight out of my marketing-technology background.

**Action.** I wrote a Python CLI: pandas for the math, openpyxl for the formatting. It produces a one-page KPI summary plus four breakdown sheets, with conditional formatting that flags any ROAS under 1.0 in red. I deliberately separated the pure calculation functions from the Excel-writing code so the math is unit-tested independently, and I baked every metric definition into the workbook so nobody has to guess how a number was computed. I also wrote a seeded generator that produces realistic but clearly-labeled synthetic data, so the whole thing runs out of the box.

**Result.** On the synthetic dataset, the read was clear: **Display was the only channel underwater** — ROAS 0.81, eating ~22% of spend and 58% of impressions for a net loss — while **Email (ROAS 3.6) and the VIP segment (converting ~2.5x the others, segment ROAS ~5) were the standouts.** The decision I'd push: pause or rework Display and redeploy that budget into Email and VIP-style audiences, while protecting Paid Search as the dependable core. The tool surfaces that in seconds, with the losing ROAS flagged red on the page.

## Likely follow-up questions

**1. "Is this real data?"**
Honest answer: the dataset is **synthetic** — generated by a seeded script, clearly labeled, with no real company's numbers in it. I shaped it to behave realistically so there's something genuine to analyze. In my own marketing-technology work I've done this kind of channel-and-segment reporting against real data — pulling from GA4 and email/SFMC campaign exports and shaping it into the same kind of summary; here I used synthetic data so the project is fully shareable and the focus stays on the tooling and the method.

**2. "Why Python instead of just doing pivot tables in Excel?"**
Repeatability. A pivot table is manual every time and easy to get subtly wrong; this runs the same way on any file with one command, with the metric definitions baked in so two people can't compute ROAS three different ways. The recurring, same-shape-every-time campaign rollups are exactly the kind of task I'd reach for this on — the more often you rebuild the same report by hand, the more a one-command tool pays for itself and removes a whole class of copy-paste errors.

**3. "Walk me through how it actually works."**
CSV in → pandas groups and aggregates → five summary tables → openpyxl writes and formats the workbook (number formats, frozen headers, ROAS color rules). Pure functions do the math and are unit-tested; the CLI just wires it together.
If they want to go deeper, the function I'd walk through is **`_safe_div`** — the helper every metric divides through. It returns `NaN` instead of infinity when the denominator is zero (e.g. a channel with no clicks or no conversions), and it works on both single numbers and whole pandas columns. That one guard is why an empty or sparse channel never blows up the report with `inf`, and it's a good example of the kind of edge case I think about up front. `channel_summary` is the other one I can speak to easily: a single `groupby("channel").agg(...)` for the sums, then ROAS/CPA/conversion-rate computed on top and sorted best-to-worst.
