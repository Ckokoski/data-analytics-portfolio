# Data-Quality Audit Toolkit

> Point it at **any** CSV. It runs a data-quality checklist and writes a
> plain-language report a non-technical manager can read, ending in one verdict:
> **PASS / REVIEW / FAIL.** **Python + pandas.**

**Bonus / range piece** — a small Python tooling project, not one of the
portfolio's core analyst headliners (see the [main README](../../README.md)).
**Self-directed; the demo data is synthetic and labeled as such.**

This is my QA instinct, written as code. In five years of marketing technology I
built QA checklists for email production — the human steps that catch a broken
link or a wrong merge field *before* it ships to a million inboxes. Before you
trust any dataset, someone should run the same kind of first-pass check. This
tool makes that check repeatable, so two people can't eyeball the same file and
reach two different conclusions about whether it's clean.

## Problem

Every analysis starts with a file someone hands you, and the first real question
is never "what does it say?" — it's "can I even trust this yet?" Missing values,
rows pasted in twice, a price column with the word "free" typed into one cell, an
age of 215: these slip through a quick glance and quietly corrupt every number
downstream. Checking for them by hand is slow, easy to skip under deadline, and
inconsistent from person to person.

## Data

Two **synthetic**, deliberately **messy** demo CSVs live in [`data/`](data/),
generated by [`generate_demo_data.py`](generate_demo_data.py) with a fixed seed
(so they're reproducible) and documented in [`data/README.md`](data/README.md):

| File | Problems baked in | Verdict it earns |
|------|-------------------|------------------|
| `messy_customers.csv` | a nearly-empty `phone` column, exact-duplicate rows, scattered missing values, duplicate `customer_id`s | **FAIL** |
| `messy_orders.csv` | text in numeric columns (`"free"`, `"two"`), out-of-range values (negative quantity, 150% discount, an absurd price typo) | **REVIEW** |

Both files are **fake** and loudly labeled as such — they exist only to give the
audit genuine problems to catch. The tool itself works on any CSV, not just
these.

## Method

[`audit.py`](audit.py) reads the CSV (everything as text first, so the *tool* —
not pandas — decides what looks numeric) and runs a checklist:

- **Row & column counts** — the basic shape of the file.
- **Per-column null rate** — share of each column that's blank, counting
  fake-blanks like `N/A`, `null`, `-` as missing too.
- **Duplicate detection** — exact-duplicate full rows, plus repeated values in a
  configurable key column (`--key customer_id`).
- **Type consistency** — columns that are *mostly* numeric but contain stray
  text (the classic "amount column with `tbd` in it"), with example bad values.
  Currency formatting like `$1,250.00` is correctly read as numeric, not flagged.
- **Out-of-range numerics** — simple, configurable min/max rules supplied as a
  JSON file (`--rules`), e.g. `{"age": {"min": 0, "max": 120}}`. Skipped cleanly
  if no rules are given.
- **A "fit for analysis?" verdict** — every finding is reduced to **PASS**,
  **REVIEW**, or **FAIL** with the *specific* reasons spelled out. The
  thresholds that draw those lines are gathered and named at the top of
  `audit.py` so a reviewer can see (and disagree with) exactly where the bar is.

It writes a Markdown report to `output/<name>_report.md` and prints a short
summary to the screen. The exit code mirrors the verdict (`0` PASS / `1` REVIEW /
`3` FAIL) so the same tool can gate an automated pipeline. The code is commented
heavily for a non-engineer — each check is a small, named function you can read
one at a time.

See the generated examples:
[`output/messy_customers_report.md`](output/messy_customers_report.md) ·
[`output/messy_orders_report.md`](output/messy_orders_report.md).

## Findings / how I'd use it

The two demo reports show the two outcomes I built this to separate. Running it on
`messy_customers.csv` returns **FAIL**: the `phone` column is 83% empty (past the
50% line), 4 of 23 rows (17%) are exact duplicates so any total would be
double-counted, `email`/`city`/`signup_date` are 9–13% blank, and four
`customer_id` values repeat — so the column I'd join on isn't actually unique.
Running it on `messy_orders.csv` returns **REVIEW**: the keys are clean and nothing
is missing, but `price` and `quantity` are numeric columns with stray text (`free`,
`TBD`, `two`), and the `--rules` check catches a negative quantity, a 150% discount,
and a `999999.99` price typo. That split is the whole point — one file I should not
build anything on yet, one I can use once I've eyeballed a handful of known cells.

That distinction is exactly the "fit for analysis?" gate I want before any real
work. The expensive mistake isn't a wrong chart — it's a *confident* chart built on
a file that was never clean: revenue inflated by duplicate rows, an average dragged
to zero because one cell said "free," a segment join silently fanned out by a
repeated ID. Those don't announce themselves in the final dashboard; they look like
real numbers. This is the same failure mode I spent five years guarding against in
marketing-tech QA — a broken merge field or bad link that passes a quick glance and
only shows up after it's in front of an audience. The fix is the same too: a
repeatable first-pass check that runs *before* anyone trusts the data, so the
go/no-go call is consistent instead of a gut feel that changes person to person.

Where I'd drop it: the moment a CSV lands and before it touches anything downstream
— before a Power BI/Excel load, before a SQL import, before I write the first
`GROUP BY`. Because the exit code mirrors the verdict (`0` PASS / `1` REVIEW / `3`
FAIL), the same command can gate an automated pipeline, not just my own desk — a
FAIL stops the load instead of quietly feeding a refresh. I'd treat **REVIEW** as
"open the report and clear each flag by hand" and **FAIL** as "fix the source, then
re-run."

On the thresholds: I think the conservative defaults are right for this *as a
gate*, and I'd keep them as the starting point rather than soften them. A 5% null
column (`NULL_RATE_REVIEW`) earning REVIEW is a feature — I'd rather be told and
dismiss it than miss it. The one knob I'd actually tune per dataset is the
out-of-range rules, since "valid" is domain-specific (a 150% discount is nonsense
for orders but a legitimate value could look extreme elsewhere) — and those already
live outside the code in a JSON `--rules` file for exactly that reason. If I were
running this against a source that's blank-by-design (optional survey fields, say),
I'd raise `NULL_RATE_REVIEW` at the top of `audit.py` and note why, rather than
train myself to ignore a flag.

What it deliberately does *not* do is judge whether the numbers are *true*. It
catches the mechanical problems — blanks, dupes, wrong types, out-of-range — but it
can't know that a correctly-formatted revenue figure is off by a decimal, or that a
plausible date is the wrong one. That's why **PASS** is worded "clear to start," not
"certified": it clears the data to be *analyzed*, and the analyst still owns whether
the result is right. Drawing that line honestly is the point — the tool removes the
dumb reasons an analysis fails so a human can focus on the real ones.

## How to run

```bash
# from this folder (05-data-quality-toolkit/)
pip install -r requirements.txt

# (the demo CSVs are already committed; regenerate them only if you want to)
python generate_demo_data.py

# audit a file -> writes output/<name>_report.md + prints a summary
python audit.py data/messy_customers.csv --key customer_id
python audit.py data/messy_orders.csv --key order_id --rules rules_example.json
```

Point it at any CSV of your own:

```bash
python audit.py path/to/your_file.csv
python audit.py path/to/your_file.csv --key id --rules your_rules.json
```

A `--rules` file is just JSON mapping column names to min/max limits:

```json
{ "age": { "min": 0, "max": 120 }, "price": { "min": 0 } }
```

### Running the tests (optional)

```bash
pip install -r requirements-dev.txt
pytest tests -v
```
