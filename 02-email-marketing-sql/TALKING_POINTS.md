# Talking Points — Email Marketing Analytics with SQL

A 2-minute interview walkthrough (STAR), plus three likely follow-ups. The tool
facts (Situation / Task / Action) are filled in; anything marked
`>>> CHRISTOPHER` is yours to complete **in your own words** — your findings and
judgment, not the tool's.

## 2-minute walkthrough (STAR)

**Situation.** Email marketing teams ask the same handful of questions over and
over, and the answers all live in the send and engagement logs: which campaigns
performed, which acquisition channels bring engaged subscribers, who's gone cold,
and how fast the list is growing. Those are SQL questions — but only if the data
is modeled cleanly and the metrics are defined the same way every time.

**Task.** I built a small, realistic email-marketing database and a library of
readable SQL queries that answer those questions end to end — as a self-directed
project to demonstrate SQL I can defend line by line, on a problem straight out of
my marketing-technology background.

**Action.** I wrote a seeded Python generator that builds a SQLite database with
three tables — subscribers, sends, and an engagement fact table of about 5,000
open/click events over a year. It's clearly-labeled synthetic data, and it's
honest by construction: I never write "open rate = 30%" anywhere — each open and
click is an individual draw from underlying probabilities, so the rates *emerge*
when you aggregate, and a click can only happen on an email that was opened, which
keeps click-to-open rate valid. Then I wrote 14 numbered SQL files, each headed
with the business question it answers and its metric definitions: open / click /
CTOR by campaign and by source; "which campaigns came in below the overall
average" using a subquery benchmark in a HAVING clause; and — the part I think
matters most for deliverability — LEFT JOINs that find the *absence* of activity
to flag subscribers with no opens or clicks in the last 90 and 180 days, the
"sunset" cohort you re-engage and eventually suppress. I kept everything at a
clear level and limited myself to exactly two window-function examples, each
flagged as a stretch, so the rest stays easy to read. A single `run_queries.py`
runs all 14 against the database, and there's a pytest smoke test that rebuilds
the DB and runs every query with zero errors.

**Result.** `>>> CHRISTOPHER:` This is what interviewers care about most — what the
data told you. Run the queries, then say in 2–3 sentences: which campaign won and
which flopped (and whether the flop was an *opens* problem or a *content/CTOR*
problem), which acquisition source brought the most engaged subscribers versus the
most dead weight, and the one re-engagement / list-hygiene move you'd push the
team to make. Keep it concrete.

## Likely follow-up questions

**1. "Is this real data?"**
Honest answer: it's **synthetic** — generated by a seeded script, clearly labeled,
with fake `@example.com` addresses and no real company's numbers. I shaped it to
behave realistically (some sources more engaged than others, a believable dormant
cohort) so there's something genuine to analyze, but every number is illustrative.
`>>> CHRISTOPHER:` optionally add one line about your *real* email/marketing-data
experience (e.g. SFMC reporting, GA4, a real ESP) so they know you've done this on
live data too.

**2. "Walk me through how you'd find subscribers to stop emailing — and why bother?"**
Tool-factual part: I LEFT JOIN every subscriber to *only* their recent engaged
rows (opened or clicked within the window); anyone with no match comes back NULL,
and filtering on `IS NULL` gives exactly the people with no recent engagement — an
INNER JOIN would silently drop the very rows I want. I do it at 90 days
(re-engage) and 180 days (suppress). The "why" is deliverability: mailbox
providers watch engagement, so continuing to email people who never open drags
down inbox placement for the *whole* list — trimming the dead weight helps
everyone who remains.
`>>> CHRISTOPHER:` add your read — roughly what share of this list is a sunset
candidate, and how aggressive you'd actually be about suppressing versus
re-engaging first.

**3. "You used window functions — can you explain one, and why you avoided them elsewhere?"**
I used exactly two, both flagged as stretch: a running total of subscribers by
month (`SUM(...) OVER (ORDER BY month)`), and `RANK() OVER (ORDER BY click_rate
DESC)` for a campaign leaderboard. A window function calculates across a set of
related rows *without* collapsing them the way GROUP BY does — so the running
total keeps each month's row while adding a cumulative column. I deliberately kept
every other query to core SQL so the project stays readable and I can defend each
line; the window functions are there to show I can reach for the right tool when it
genuinely helps.
`>>> CHRISTOPHER:` pick **one** query file you're most comfortable explaining
cold — query 05 (the subquery-benchmark HAVING) or query 10 (the LEFT JOIN sunset
logic) are good choices — and rehearse walking through it, so it isn't cold if
they ask you to go deeper.
